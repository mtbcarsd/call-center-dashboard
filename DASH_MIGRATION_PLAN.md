# Миграция Streamlit → Dash + ролевые дашборды

Составлено 2026-07-21 по запросу пользователя — план на несколько будущих сессий,
пока не начатый. Источник правды по общему бэклогу — README.md → «Куда развиваться»
(там будет короткая ссылка на этот файл); здесь — детальный план именно этой задачи.

## Контекст

Дашборд сейчас — один Streamlit-файл (`dashboard.py`, ~906 строк) с единственным
хардкоженным пользователем-админом (`USERS` dict, `dashboard.py:33-39`), без какого-либо
разделения по ролям — `role` хранится в сессии, но нигде не используется для
фильтрации данных (`dashboard.py:242-243`).

Пользователь хочет:
1. Переехать на **Plotly Dash** — мотив: показать красивые графики самому главному
   начальнику (не техническая жалоба на Streamlit, а презентационная задача).
2. Три уровня дашбордов:
   - **Директор** — полный обзор по всем отделам (как нынешний admin-вид, но
     визуально более сильный).
   - **Начальник отдела** — видит только свой отдел и его сотрудников.
   - **Сотрудник** — видит только свою историю звонков/статистику + **рекомендации
     по обучению** (новая фича, которой нет вообще).

Важно: пункт «персональный кабинет оператора» **уже был в бэклоге проекта**
(README.md, Phase 5 из анализа конкурента MeliSSA) — просто не был реализован.
Мы не изобретаем требование, а достраиваем то, что уже отложено, плюс добавляем
слой «начальник отдела» и переезд на Dash.

**Подтверждённые пользователем решения:**
- Когда Dash догонит Streamlit по функционалу (включая вкладку «Звонки» с
  редакторами/плеером) — переключаемся. Таблицы с прогресс-барами (аналог
  `st.column_config.ProgressColumn`) — через **dash-ag-grid**.
- **Обновлено 2026-07-24 (D5):** решение по отключению Streamlit смягчено —
  вместо полного отключения выбран мягкий cutover: Dash становится основным
  публичным дашбордом, Streamlit-сервис остаётся живым на своём URL как
  подстраховка для отката, без установленного срока отключения (см. Phase D5
  ниже).

## Что уже проверено в коде (не гадаем, а исходим из фактов)

- `operator_name` (`db.py:97-100`) — свободный TEXT, без FK на отдельную таблицу
  операторов, ставится вручную через текстовое поле (`dashboard.py:502-521`). Нет
  вообще никакой связи логин → operator_name/department.
- Полная схема `call_analysis` (`db.py:11-107`): `department, call_topic,
  transcript_text, call_summary, call_type/call_type_override, urgency,
  customer_intent, resolution_status, agent_performance_score, customer_satisfaction,
  qa_score, escalation_flag, key_topics, checklist_json, compliance_json,
  action_items_json, operator_name, call_datetime, analyzed_at, audio_key,
  silence_*, pause_count, operator_talk_ratio`. Плюс `call_segments`,
  `tags`/`call_tags`, `collections`/`call_collections`, `comments`.
- Разбор `checklist_json`/`compliance_json` сейчас живёт **внутри `dashboard.py`**
  (`_parse_checklist`, `_checklist_pass_rates` — строки 701-716; `_parse_compliance`
  — строки 770-777) — нужно вынести в `checklist.py`, чтобы переиспользовать и в
  Dash-приложении, и в новом coaching-агенте.
- `agents/trends.py` + `GET /trends` (`api/main.py:95-116`) — ближайший аналог для
  будущего per-operator coaching-агента: тот же `BaseAgent`, но агрегирует по ВСЕМ
  звонкам, а не по одному оператору.
- Депло: `api/Dockerfile` — рабочий самодостаточный паттерн (в отличие от
  Streamlit-сервиса, который живёт на Nixpacks + ручной старт-командой в Railway UI,
  после инцидента с `railway.toml`, коммит `1418e6b`). Новый Dash-сервис — по образцу
  `api/Dockerfile`, не по образцу Streamlit-сервиса.

## Изменения в данных

**Новая таблица `users`** (реальная таблица, не dict — Dash/Flask должен проверять
логин на каждый запрос, а не держать это в коде):

```sql
CREATE TABLE IF NOT EXISTS users (
    username TEXT PRIMARY KEY,
    password_hash TEXT NOT NULL,
    role TEXT NOT NULL,             -- 'executive' | 'manager' | 'operator'
    display_name TEXT,
    department TEXT,                -- обязательно для role='manager'
    operator_match_name TEXT,       -- обязательно для role='operator', сверяется с call_analysis.operator_name
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

- Добавляется тем же паттерном `CREATE TABLE IF NOT EXISTS`, что и остальные таблицы
  в `db.py`, без ORM.
- **Без FK** `users.operator_match_name` → `call_analysis.operator_name` — сохраняем
  текущий принцип (сравнение по точному совпадению строки, нормализованному —
  `.strip().lower()` — на момент фильтрации, а не жёсткая связь). Это тот же дух, что
  и решение "не делать таблицу operators при 11 звонках" — просто теперь под логин
  нужно ГДЕ-ТО хранить credentials, чего раньше не требовалось вообще.
- **Хэш пароля: переход на `bcrypt`/`passlib`**, а не текущий голый `hashlib.sha256`
  без соли (`dashboard.py:41-45`). Раньше это был один захардкоженный пароль админа;
  теперь — публичная форма логина на Flask с потенциально 10+ реальными учётками
  (операторы + начальники отделов) — риск другого порядка, стоит закрыть сразу, а не
  переносить старую схему как есть.
- Одноразовый скрипт-миграция (по образцу `scripts/add_new_calls.py`,
  `INSERT ... ON CONFLICT DO NOTHING`) переносит текущего пользователя Julia в
  `users` с `role='executive'`.

## Фича «рекомендации по обучению» (coaching)

- **`agents/coaching.py`** — новый агент на основе `BaseAgent` (как и `trends.py`),
  но на вход берёт историю ОДНОГО оператора (чек-лист по каждому из 6 пунктов
  `checklist.py`, QA-оценки и их расхождение с AI-оценкой, compliance-нарушения), а
  не агрегат по всем звонкам. Промпт просит сильные/слабые стороны и конкретные
  рекомендации — только по этому оператору, без сравнения с другими.
- **`GET /coaching/{operator_name}?limit=20`** в `api/main.py` — структурно как
  существующий `GET /trends` (запрос к Postgres + вызов агента), сохраняя принцип
  проекта «дашборд не тянет ML/LLM-зависимости, только HTTP до `api`».
- Синхронный вызов по образцу `fetch_trends()`/кнопки обновления (`dashboard.py:20-24,
  860-861`), не отдельная таблица с фоновым пересчётом — при текущем объёме звонков
  на оператора это лишнее, добавим позже, если станет реально медленно/дорого.

## Стратегия миграции Streamlit → Dash

**Переносится почти бесплатно**: все объекты `go.Figure`/`px.bar`/`px.pie`
(`dashboard.py:320-369`) — Dash и есть фреймворк Plotly, меняется только обёртка
(`st.plotly_chart(fig)` → `dcc.Graph(figure=fig)`). Вкладки Операторы/Рейтинг/
Compliance/Тренды — простой `groupby` → таблица, без Streamlit-специфичного
состояния.

**Требует настоящей переработки** (декларативный layout + callback'и вместо
императивного rerun):
- **Аутентификация** — у Dash нет встроенного логина; это уровень Flask
  (`app.server`): страница логина, `session` на подписанной cookie,
  `@server.before_request` для защиты роутов. Логика хэш-проверки переносится
  почти как есть, обвязка (сессия/редирект/защита роутов) — новый код.
  **Критично**: для начальника отдела фильтр по department должен применяться
  НА СЕРВЕРЕ внутри callback/запроса из `session["department"]`, а не просто
  скрываться в UI — иначе прямой вызов callback обходит ограничение.
- **Таблицы с прогресс-барами** → `dash-ag-grid` (подтверждено пользователем).
- **KPI-карточки** (`st.metric`) → свой `stat_tile()`-компонент — сюда же основной
  фокус визуального полиша под "показать директору" (используем skill `dataviz`
  для палитры/формы).
- **Ролевая маршрутизация** — через `dash.register_page`/`pages/`, не через
  вкладки в одном файле.
- **Вкладка «Звонки» (самая тяжёлая)**: галерея карточек, инлайн-редакторы
  (`call_type_override`/`operator_name`/`qa_score`, сейчас —
  `st.session_state[f"editing_..._{file_name}"]`, `dashboard.py:496-521`),
  мультиселект тегов/коллекций, комментарии — всё переезжает на
  pattern-matching callbacks (`MATCH`/`ALL` по `id={"type": ..., "index": file_name}`)
  и `dcc.Store` вместо `session_state`.
- **Аудиоплеер с перемоткой по клику** (`st.audio(url, start_time=X)`,
  `dashboard.py:446`) — у Dash нет прямого аналога; нужен `html.Audio` +
  `clientside_callback` (JS в браузере), который выставляет
  `audioEl.currentTime = seekTime` при изменении `dcc.Store`. Это по-настоящему
  новый код, не перенос.

**Тестирование**: аналога `AppTest` в Dash нет (только `dash.testing`/Selenium,
тяжелее). Основной уровень — юнит-тесты callback-функций напрямую (как уже
делается в `tests/test_pipeline.py` для `orchestrator.analyze()`); отдельный,
редко запускаемый Selenium-набор — только для перемотки аудио и MATCH-флоу
редакторов (по аналогии с `RUN_SLOW_TESTS=1`).

## Деплой на Railway

- Новый `dash_app/Dockerfile` по образцу `api/Dockerfile` (самодостаточный, не
  полагается на общий Nixpacks-конфиг, который уже один раз ломал старт-команды
  других сервисов).
- `CMD ["sh", "-c", "gunicorn dash_app.app:server --bind 0.0.0.0:$PORT --workers 2"]`
  — `sh -c` обязателен, иначе `$PORT` не подставится.
- **Отдельный `requirements-dash.txt`**, не трогаем текущий `requirements.txt`
  Streamlit-сервиса (там уже трижды ловили сегфолт из-за непиненных
  pandas/pyarrow/psycopg2) — каждая новая зависимость пинится явно с первого дня.
- Multi-worker gunicorn ≠ Streamlit per-session state: воркеры не шарят память
  между собой. Кэш HTTP-ответов (аналог `@st.cache_data`) — либо `workers=1` на
  старте, либо `flask_caching.SimpleCache` (per-worker кэш ок при текущем объёме
  данных).
- Новый Railway-сервис разворачивается параллельно на всё время разработки (это
  не отдельное решение — так и так нет паритета до фазы D4). После достижения
  паритета — переключаем публичный домен на Dash и отключаем Streamlit-сервис
  (оставить доступным по авто-URL на короткое окно отката перед полным удалением).

## Фазы (независимо деплоятся и тестируются)

- D1.1 Каркас `dash_app/` (`app.py`, `pages/`, `requirements-dash.txt`, `Dockerfile`),
  новый Railway-сервис параллельно (пока без кастомного домена). Перенос вкладки

- D1.2 Остальные read-only вкладки как Dash Pages: Операторы, Рейтинг, Compliance,

- D1.3 Полиш под директора: `stat_tile()`, `dash-ag-grid` для таблиц, единая

- D1.4 Юнит-тесты callback-функций.

### Phase D2 — Логин + роли + начальник отдела
- D2.1 Таблица `users` + миграция Julia → `role='executive'`.
- D2.2 Страница логина на Flask-сессии + `before_request`-гейт, логаут.

- D2.4 Деплой + тест: начальник одного отдела не видит данные другого (включая

### Phase D3 — Личный кабинет сотрудника + рекомендации по обучению
- D3.1 Страница сотрудника: звонки, отфильтрованные по

### Phase D4 — Вкладка «Звонки» (самое тяжёлое)

- D4.3 Инлайн-редакторы (категория/оператор/QA-оценка) через `MATCH`/`ALL`

- D4.5 Тесты: юнит на callback-запись в БД (основные), отдельный редкий

## Проверка (для каждой фазы)

- Юнит-тесты callback-функций (вызывать напрямую как обычные Python-функции с
  тестовыми данными/подменённым слоем БД), как уже делается для `orchestrator.analyze()`
  в `tests/test_pipeline.py`.
- Ручная проверка через реальный браузер на новом Railway-сервисе (URL из
  `railway domain`/логов деплоя) — особенно для D1 (визуальный чекпойнт для
  демонстрации) и D4 (перемотка аудио, редакторы).
- Для D2-D3 — явный тест на утечку данных между ролями (начальник/сотрудник не
  видит чужое) не только через UI, но и прямым вызовом callback с чужими
  параметрами.
- Деплой по уже отработанному в проекте паттерну: `railway up --service <name>
  --environment production --detach`, опрос `railway deployment list` до
  терминального статуса, проверка логов на чистоту (без сегфолтов/трейсбеков).
