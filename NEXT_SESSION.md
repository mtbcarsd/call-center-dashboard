## Текущий статус (коротко)

Полная картина — в README.md → «Статус деплоя» и «Куда развиваться». Тут только то,
что нужно держать в голове перед тем как начинать новую задачу:

- **Локально (`docker-compose up`)** — весь стек работает без ограничений: API,
  OpenWebUI-чат, batch-pipeline, Grafana, Postgres.
- **Railway — план проекта апгрейднут на Hobby (2026-07-18, сессия 8)**. RAM больше
  не блокер: `api` передеплоен и работает чисто (`SUCCESS`, без OOM). `GET /trends`
  проверен end-to-end (реально ходит в Groq, возвращает содержательный ответ) —
  **пункт 3.4 больше не заблокирован инфраструктурно**, можно возвращаться к нему.
  LLM-бэкенд для `api`/`pipelines` — Groq (`LLM_BASE_URL=https://api.groq.com/openai/v1`),
  облачный, от RAM не зависит.
- **`pipelines` (OpenWebUI-чат) — всё ещё НЕ РАБОТАЕТ**, но по ДРУГОЙ причине, не RAM:
  `ghcr.io/open-webui/pipelines:main` несёт свой набор torch+torchvision+torchaudio,
  несовместимый с нашим `torch==2.12.1` из `requirements-ml.txt`. Нашли и вживую
  (через `railway ssh`) подтвердили фикс — снести весь базовый torch-стек и дать pip
  пересобрать зависимости `pyannote.audio` (включая torchaudio) с нуля одним проходом
  (см. `pipelines/Dockerfile`, закомментировано). Но после передеплоя с этим фиксом
  **в реальном контейнере `torchvision` всё равно на месте** (сиротой, без
  Required-by), хотя build-логи чётко показывают чистую установку без него —
  необъяснённое расхождение build vs runtime. Гипотезы про frontmatter-триггер
  фреймворка и start.sh проверены и отклонены. **Сознательно остановились** — чат не
  в приоритете, а дальнейший поиск причины даёт всё меньше отдачи. Если возвращаться:
  вероятно нужен `railway ssh` (ключ уже зарегистрирован на аккаунте, см. `railway ssh
  keys list`) и, может быть, полный отказ от базового образа `ghcr.io/open-webui/
  pipelines:main` в пользу чистого `python:3.11-slim` (как у `api/Dockerfile`) +
  установка самого framework'а поверх — это не потребует бороться с чужим
  преднастроенным torch-стеком вообще.
- **Streamlit-дашборд** — Phase 1, 2 и 3 (кроме отложенного 3.4) сделаны и
  задеплоены. Есть особенность: **любая правка requirements.txt без явного пина
  версии рискует сегфолтом** — на этом стеке уже трижды ловили сегфолт из-за
  неопределённых версий pandas/pyarrow/psycopg2 (см. git log, коммиты про "Fix
  segfault"/"Pin pyarrow"). Перед добавлением новой зависимости — пинить версию явно.
- **Данные**: 11 звонков (5 ОО + 5 ОРККиП + 1 CARtest), синхронизированы в локальной
  и Railway Postgres, аудио для всех залито в bucket. Все 10 звонков в call_analysis
  имеют один и тот же `analyzed_at` (один batch-прогон пайплайна) — если планируете
  что-то по временным трендам, это ограничение актуально.
- **Roadmap**: README.md → «Куда развиваться» — Phase 1-3 done (3.4 отложен),
  Phase 4-5 — списки идей, ещё не расписанные по шагам (см. ниже).

## Фазы 1-3 — ✅ ЗАВЕРШЕНЫ (что сделано)

**Phase 1:** карточная галерея звонков, аудиоплеер, click-to-seek по репликам,
S3-хранилище аудио (Railway Bucket).

**Phase 2 — организация данных** (сессия 8, 2026-07-18):
- **2.1** Ручная валидация AI-категорий — `call_type_override`, ✅ Подтвердить/❌ Исправить.
- **2.2** Дикторы — `operator_name`, вкладка «Операторы» со статистикой.
- **2.3** Теги и коллекции — `tags`/`call_tags`/`collections`/`call_collections`,
  редактор в деталке, фильтры в сайдбаре.
- **2.4** Комментарии — таблица `comments`, список + форма под транскриптом.

**Phase 3 — то, что уже посчитано, но не показано** (сессия 8, 2026-07-18):
- **3.1** Вкладка «🏆 Рейтинг» — % прохождения по каждому пункту чек-листа (без
  изменений схемы, использует существующий `checklist_json`), худшие сверху,
  разбивка по оператору. Нашли реальный паттерн: «Выявление потребности» false во
  всех 10 звонках — подсвечено предупреждением в UI.
- **3.2** Вкладка «🛡️ Compliance» — % прохождения, список звонков с нарушениями,
  разбивка по оператору (без графика «по времени» — все звонки из одного batch).
- **3.3** Ручная QA-оценка — `qa_score REAL` (nullable), редактируется в деталке
  рядом с AI-оценкой, показывает расхождение (Δ), добавлена в таблицу «Операторы».
- **3.4 (вкладка «Тренды» через `agents/trends.py`) — БОЛЬШЕ НЕ ЗАБЛОКИРОВАНА**
  (обновление той же сессии, после апгрейда на Hobby): `api` теперь работает, `GET
  /trends` проверен end-to-end. Реализация: дашборду нужен HTTP-клиент к публичному
  `api-production-95c7e.up.railway.app/trends` (`requests` уже не в requirements.txt
  дашборда — придётся добавить и явно запинить версию, см. правило про сегфолты
  выше). Это лёгкая зависимость, никакого `openai`/ML-стека в дашборд тащить не
  нужно — вся LLM-логика уже посчитана на стороне `api`.

Все пункты проверены `AppTest` (клики по виджетам, headless) + прямой проверкой
записи/отката в Postgres, задеплоены на Railway (`SUCCESS`, прод 200, логи чистые).
Подробности каждого пункта, если нужны, — в `memory_session.md` (память проекта),
сессия 8.

**Важный урок на будущее** (пригодится для любых новых виджетов с БД-состоянием):
если `st.multiselect`/`st.text_area`/другой виджет с `key` одновременно управляется
и пользовательским вводом, и внешним состоянием (БД), то `default`/`value` не
подхватится на повторных рендерах, если `key` не меняется — Streamlit держит старое
значение виджета из session_state. Фикс — включать состав актуальных данных в сам
`key` (см. `render_label_editor` и текстовое поле комментариев в `dashboard.py`).
