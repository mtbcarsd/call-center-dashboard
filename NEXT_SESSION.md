## Текущий статус (коротко)

Полная картина — в README.md → «Статус деплоя» и «Куда развиваться». Тут только то,
что нужно держать в голове перед тем как начинать новую задачу:

- **Локально (`docker-compose up`)** — весь стек работает без ограничений: API,
  OpenWebUI-чат, batch-pipeline, Grafana, Postgres.
- **Railway** — `call-center-dashboard` (Streamlit), `openwebui`, `Postgres`,
  `call-audio` (bucket) — работают. `api` и `pipelines` — **CRASHED/FAILED**, упираются
  в 1GB RAM Trial-плана (torch+whisper+pyannote не влезают). Апгрейд на Hobby отложен
  пользователем на неопределённый срок — не предлагать сделать это самостоятельно,
  просто иметь в виду при вопросах про доступность чата на проде.
- **Streamlit-дашборд** — Phase 1 сделана (галерея звонков, аудиоплеер, click-to-seek
  по репликам, S3-хранилище аудио). Есть особенность: **любая правка requirements.txt
  без явного пина версии рискует сегфолтом** — на этом стеке уже трижды ловили
  сегфолт из-за неопределённых версий pandas/pyarrow/psycopg2 (см. git log, коммиты
  про "Fix segfault"/"Pin pyarrow"). Перед добавлением новой зависимости в
  requirements.txt — пинить версию явно, а не оставлять пустой.
- **Данные**: 11 звонков (5 ОО + 5 ОРККиП + 1 CARtest), синхронизированы в локальной
  и Railway Postgres, аудио для всех залито в bucket.
- **Roadmap**: README.md → «Куда развиваться» — Phase 1 done, Phase 2 next (план ниже),
  Phase 3-5 расписаны, но не в приоритете, пока не закрыта Phase 2.

## План Фазы 2 (следующая задача)

Три независимых куска, можно делать по одному, порядок — по возрастанию сложности.

### 2.1 — Ручная валидация AI-категорий — ✅ СДЕЛАНО (сессия 8, 2026-07-18)

- `call_analysis.call_type_override TEXT` (nullable), миграция в `db.py` через
  `_NEW_CALL_ANALYSIS_COLUMNS`.
- Деталка звонка: ✅ Подтвердить (фиксирует текущее AI-значение как override) /
  ❌ Исправить (текстовое поле → Сохранить). `call_type_effective` (override, если
  задан, иначе AI-значение) используется везде: карточки галереи, таблица «Все
  звонки», деталка. 🏷️ — маркер, что категория подтверждена человеком.
- Отдельный компонент истории изменений сознательно не делали (это Phase 5, избыточно
  для 11 звонков) — как и планировалось.
- Проверено `AppTest` (логин → открыть звонок → подтвердить → изменить → сохранить,
  без исключений) + прямая проверка записи/отката в Postgres. Задеплоено на Railway
  (`railway up --service call-center-dashboard`), деплой `SUCCESS`, прод отвечает 200,
  в логах контейнера чисто (без сегфолтов).

### 2.2 — Дикторы (именование операторов) — ✅ СДЕЛАНО (сессия 8, 2026-07-18)

- `call_analysis.operator_name TEXT` (nullable), миграция в `db.py` через
  `_NEW_CALL_ANALYSIS_COLUMNS`. Обычное текстовое поле, без отдельной таблицы
  `operators` с FK — как и планировалось, избыточно при 11 звонках.
- Деталка звонка: ✏️ Указать/Изменить оператора → текстовое поле → Сохранить/Отмена
  (тот же паттерн редактирования, что и у `call_type_override` в 2.1). Карточки
  галереи показывают 🧑‍💼 имя, если оно проставлено.
- Новая вкладка «🧑‍💼 Операторы»: таблица `operator_name → звонков, оценка
  (чек-лист), удовл. клиента` — по звонкам, где имя уже указано, плюс счётчик
  ещё неразмеченных.
- Проверено `AppTest` (логин → открыть звонок → указать оператора → сохранить →
  проверить отображение → открыть вкладку статистики, без исключений) + прямая
  проверка записи/отката в Postgres. Задеплоено на Railway, деплой `SUCCESS`,
  прод отвечает 200, логи чистые.

### 2.3 — Теги и коллекции

- Схема: `tags(id, name)`, `call_tags(file_name, tag_id)`, `collections(id, name)`,
  `call_collections(file_name, collection_id)` — обычные many-to-many таблицы,
  `CREATE TABLE IF NOT EXISTS` в `db.py`.
- UI: мультиселект тегов на карточке звонка (с возможностью вписать новый тег —
  `st.multiselect` не поддерживает "создать на лету" сам по себе, нужен доп. text_input
  + кнопка "добавить тег"). Коллекции — второй, отдельный мультиселект того же вида.
- Фильтр по тегу/коллекции в сайдбаре — рядом с существующими фильтрами
  Отдел/Срочность/Статус.

### 2.4 — Комментарии

- Схема: `comments(id SERIAL, file_name TEXT, author TEXT, text TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP)`.
- UI: под транскриптом на карточке звонка — список существующих комментариев
  (автор + дата + текст) и форма добавления нового (`st.text_area` + кнопка).
  `author` = `st.session_state["username"]` (у нас уже есть логин).

### Общее для всех четырёх

- Каждый пункт — это: миграция в `db.py` → правка `dashboard.py` (вкладка «Звонки») →
  локальная проверка через `AppTest` (см. как это делалось в Phase 1 — клик по кнопкам,
  проверка на исключения) → коммит → `railway up --service call-center-dashboard`.
- Один и тот же паттерн деплоя, что и в Phase 1 — `railway up --service <name>
  --environment production --detach`, потом опрос `railway deployment list` до
  терминального статуса (**не** докладывать об успехе по одному "Build queued").
