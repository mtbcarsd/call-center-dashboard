**Важный урок, пригодится для будущих виджетов с БД-состоянием:** если
`st.multiselect`/`st.text_area`/другой виджет с `key` одновременно управляется и

значение виджета из session_state. Фикс — включать хэш/состав актуальных данных в
сам `key` (см. `render_label_editor` и `add_comment`-текстовое поле в `dashboard.py`).

## План Фазы 3 (следующая задача)

Код-ревью подтвердил актуальность `checklist.py`/`agents/trends.py`/`GET /trends`, но
также вскрыл два важных факта о реальных данных, которые определяют план:

- **На Railway нет рабочего LLM-бэкенда**: `api` и `pipelines` оба CRASHED (тот же
  OOM), а `openwebui` настроен ходить за LLM именно в `pipelines.railway.internal:9099`
  — то есть чат и любые новые LLM-вызовы на проде сейчас не работают вообще, не
  только транскрипция/диаризация.
- **Все 10 звонков в базе имеют один и тот же `analyzed_at`** (один batch-прогон
  09.07.2026) — «тренды по времени» на этих данных технически будет одной точкой.
- Полезная находка при агрегации `checklist_json`: пункт «Выявление потребности»
  (`needs_discovery`) — **false во всех 10 звонках без исключения**. Стоит подсветить
  в UI 3.1 (не просто сухая таблица).

Три пункта, по возрастанию сложности, **3.4 сознательно отложен** (см. ниже):

- **Реальная находка**: пункт «Выявление потребности» (`needs_discovery`) — false
  во всех 10 звонках в базе. Предупреждение в UI это подсвечивает.
- Проверено `AppTest` (дефолтное состояние + отдельно с временно проставленным
  `operator_name`, откачено). Задеплоено на Railway, `SUCCESS`, прод 200, логи чистые.

### 3.2 — Compliance-вкладка — ✅ СДЕЛАНО (сессия 8, 2026-07-18)
